      Tabular Comparison of Key SQE Concepts

      Side-by-side analytical comparisons of essential methodologies and frameworks for quick exam revision.

      1. Static vs. Dynamic Testing

      Feature Static Testing Dynamic Testing
      Execution Done without running the code (inspection, walkthrough, review, static analysis). Done by executing the software with test cases.
      Goal Prevent defects early by finding logic errors, syntax flaws, or deviations from standards. Identify bugs, performance bottlenecks, and validate functional compliance by observing behavior.
      Stage Performed early in the SDLC (requirements, design, and code review). Performed later in the SDLC (integration, system, and user acceptance testing).
      Cost of Fix Very low (fixed before implementation). Relatively high (requires debugging, re-coding, and re-testing).

      2. White Box vs. Black Box vs. Experience-Based Testing

      Feature White Box Testing Black Box Testing Experience-Based Testing
      Code Visibility Complete visibility of internal source code, structure, and logic. No visibility of internal code; system is treated as a black box. Based on tester's knowledge, intuition, skills, and past experience.
      Tester Persona Developers or specialized automation/white-box testers. Independent testing teams, business analysts, or end users. Experienced testers who understand domain, user behaviors, or technology stack.
      Techniques Statement coverage, Branch coverage, Path coverage, Data flow testing. ECP, BVA, Decision Table Testing, State Transition, Use Case Testing. Error Guessing, Exploratory Testing, Checklist-Based Testing.
      Focus Code execution completeness, dead code removal, logical path validation. Functional requirements, input-output correctness, external interfaces. Uncovering unanticipated bugs, edge cases, and real-world usage patterns.

      3. Error Guessing vs. Exploratory Testing

      Feature Error Guessing Exploratory Testing
      Definition An intuitive technique where the tester anticipates common developer mistakes based on experience. An unstructured hands-on approach where test design, execution, and learning happen simultaneously.
      Rigor & Formality Highly informal; relies heavily on unstructured intuition and custom checklists. Semi-structured; often uses session-based test management (charters, time-boxes, logs).
      Key Driver Historical defect patterns, typical developer errors (e.g., divide by zero, null values). Active learning of the system under test, curiosity, and rapid feedback loop.

      4. Verification vs. Validation

      Feature Verification Validation
      Core Question "Are we building the product right?" "Are we building the right product?"
      Execution Does not involve software execution (mostly static evaluation). Involves executing the final software product (mostly dynamic evaluation).
      Focus Ensures specifications, requirements, architecture documents, and designs are met. Ensures the product matches the customer's actual operational needs and expectations.
      Activities Reviews, Walkthroughs, Inspections, Desk-checking. System testing, User Acceptance Testing (UAT), Beta testing.

      5. Severity vs. Priority

      Feature Severity Priority
      Definition Refers to the technical impact of a defect on the system's operation/functionality. Refers to the business urgency of fixing a defect in terms of scheduling and release timeline.
      Set By Assessed by QA engineers / Testers based on technical system impact. Determined by Product Owner / Project Manager / Business Analyst.
      Examples High Severity, Low Priority: A minor typo in a critical legal document or system crash on a rare edge case.
      Low Severity, High Priority: A wrong logo on the landing page of a major release.      		 High Priority, High Severity: A database connection failure on the main login screen.

      6. QA vs. QC

      Feature Quality Assurance (QA) Quality Control (QC)
      Focus Process-oriented: ensuring that processes and standards are followed to build a quality product. Product-oriented: inspecting and testing the product to find and fix defects.
      Approach Proactive: aimed at defect prevention before they are introduced. Reactive: aimed at defect detection and correction after they are introduced.
      Responsibility Entire project team (managers, developers, SQA team). Dedicated testing team / QA engineers.

      7. Product vs. Process vs. Project Metrics

      Feature Product Metrics Process Metrics Project Metrics
      Measurement Domain Measures the attributes of the software product itself (code, design, specifications). Measures the attributes of the software development and QA processes. Measures progress, resources, costs, schedules, and status of the project.
      Purpose Assess software quality, size, complexity, and maintainability. Evaluate effectiveness of development, testing, and defect-removal activities. Manage scope, budget, schedule risks, and resource utilization.
      Examples Lines of Code (LOC), Cyclomatic Complexity, Defect Density, Problems per User Month (PUM). Defect Removal Effectiveness (DRE), Defect Injection Rate, Prep Effort. Sprint Velocity, Budget variance, Milestone schedule slips, resource turnover.

      8. MTBF vs. MTTF vs. MTTR

      Feature Mean Time Between Failures (MTBF) Mean Time To Failure (MTTF) Mean Time To Repair (MTTR)
      System Type Applies to repairable systems. Applies to non-repairable systems (e.g. spacecraft parts, light bulbs). Applies to repairable systems.
      Definition The average time elapsed between consecutive failures of a system during active service. The average time a system operates continuously before encountering its first/only failure. The average time required to troubleshoot, fix, and restore a failed system to operations.
      Formula Relationship $$MTBF = MTTF + MTTR$$ (often simplified to $$MTBF \approx MTTF$$ when MTTR is negligible). $$MTTF = \frac{\text{Total operating hours}}{\text{Total number of failures}}$$ $$MTTR = \frac{\text{Total downtime spent in repairs}}{\text{Total number of failures}}$$

      9. Load vs. Stress vs. Volume Testing

      Feature Load Testing Stress Testing Volume Testing
      Primary Focus Validating system performance and response times under expected standard and peak usage. Determining system robustness and failure recovery when pushed beyond resource limits. Verifying database stability and system performance when handling extremely large datasets.
      Key Goal Identify response time delays, database bottlenecks, and capacity thresholds. Find the system breaking point, verify clean failover, and prevent data corruption. Check for memory leaks, slow indexing, database locks, and query timeouts under mass data.
      Test Scenario Simulating 1,000 concurrent user requests on an e-commerce website over a 2-hour sale. Simulating a sudden spike of 50,000 concurrent users (DDoS-like) to test crash recovery. Uploading a 10 million record customer CSV file to audit database storage limits.

      10. Unit vs. Integration vs. System Testing

      Feature Unit Testing Integration Testing System Testing
      Scope Verifying small, isolated components of code (like individual functions, classes, or modules). Testing communication interfaces and interactions between integrated subsystems or modules. Evaluating the complete, fully integrated software application against final requirements.
      Performed By Software Developers. Developers or Integration QA Engineers. Independent Testing / QA Teams.
      Environment Highly mocked (using stubs, mocks, and virtual dependencies). Hybrid environment combining real subsystems and mocked external APIs. Production-like staging environment with real databases and hardware.

      11. Alpha vs. Beta Testing

      Feature Alpha Testing Beta Testing
      Location Performed inside the development organization's lab environment. Performed in the real-world operational environment of the end users.
      Testers Internal employees, developers, and QA staff. Actual customers, target users, or a selected external pilot group.
      Main Goal Identify major functional flaws and showstopper bugs before wider external visibility. Gather real-world feedback on usability, reliability, compatibility, and edge cases.

      12. Automated vs. Manual Testing

      Feature Manual Testing Automated Testing
      Execution Human testers execute test cases manually, verifying outcomes step-by-step. Software test scripts and frameworks run tests and compare outputs automatically.
      Suitability Best for exploratory, ad-hoc, usability, and initial one-off test runs. Best for regression testing, performance load testing, and highly repetitive tests.
      Initial Cost Low (only requires test case documentation). High (requires test automation framework setup and script writing).
      Run Speed Slow, limited by human interaction speeds. Extremely fast, can run concurrently and in background pipelines.

      Quick SQA Formula Matrix

      Metric Formula Standard / Threshold / Purpose
      DRE (Defect Removal Effectiveness) $$DRE = \frac{E}{E + D}$$ Ideal: > 85%. E = pre-release bugs, D = post-release bugs.
      Cyclomatic Complexity $$V(G) = E - N + 2P$$ or $$P + 1$$ Keep under 10. E = edges, N = nodes, P = components.
      Rework Ratio $$\frac{\text{Rework Hours}}{\text{Total Hours}} \times 100$$ Goal: < 10-15% of total development effort.
      MTBF (Mean Time Between Failures) $$MTBF = MTTF + MTTR$$ Higher is better. Measures average uptime between crashes.
      Availability $$\frac{MTBF - MTTR}{MTBF} \times 100$$ Target: > 99.9% (Three Nines uptime).
      PUM (Programmer Productivity) $$\frac{\text{LOC}}{\text{Person-Month}}$$ Measures throughput volume. Not a quality indicator.
      CPU Execution Time $$Time = IC \times CPI \times Clock$$ IC = Instruction Count, CPI = Cycles Per Instruction.
      Defect Density $$\text{Density} = \frac{\text{Defects}}{\text{KLOC (or FP)}}$$ Measures software quality density. Lower is better.
      Unadjusted Function Points $$UFP = \sum (\text{Count} \times \text{Weight})$$ Counts Inputs, Outputs, Inquiries, Files, Interfaces.
      Test Case Effectiveness $$TCE = \frac{\text{Defects found by tests}}{\text{Total defects found}} \times 100$$ Higher is better. Evaluates test suite quality.
      COCOMO Effort (Basic) $$Effort = a \times (\text{KLOC})^b$$ a, b are constants based on project complexity. Unit: Person-Months.
      COCOMO Dev Time (Basic) $$T_{dev} = c \times (Effort)^d$$ Estimates calendar duration in months.
      Schedule Slippage $$\frac{T_{actual} - T_{planned}}{T_{planned}} \times 100$$ Percentage deviation from baseline project schedule.
      KSSI (Kilo Source Statements of Instruction) $$KSSI = \frac{\text{SSI}}{1000}$$ SSI = Source Statements of Instruction. Volume metric excluding comments/whitespaces.
      COCOMO Staffing $$\text{Staff} = \frac{Effort}{T_{dev}}$$ Calculates average full-time engineers needed for project duration.
      BVA Test Cases (2-Value & Robust) $$4N + 1$$ (standard) / $$6N + 1$$ (robust) N = Number of input fields. Tells total boundary value cases to write.
      Halstead Program Volume $$V = (N_1 + N_2) \log_2 (n_1 + n_2)$$ $n_1, n_2$ = unique operators/operands. $N_1, N_2$ = total occurrences.

      SQE Revision - Printable Cheat Sheet & Formulas

      1. Defect Removal Effectiveness (DRE)

      $$DRE = \frac{E}{E + D}$$

      What Symbols Mean: E = Defects found before release (during testing), D = Defects found after release (by clients).

      Where to Use & Explanation: Evaluates how effective your quality assurance process is at catching defects before they reach users. A target DRE is > 85%.

      Roman Urdu Explanation: Yeh metric batata hai ke humne release se pehle kitne defects pakre. Jitni zyada percentage hogi, utna hi acha hamara testing process hai.

      2. Cyclomatic Complexity

      $$V(G) = E - N + 2P \quad \text{or} \quad V(G) = P + 1$$

      What Symbols Mean: E = Edges in CFG, N = Nodes in CFG, P = Connected Components (usually 1), P (in second formula) = Predicate nodes (decision points).

      Where to Use & Explanation: Measures the complexity of a program's control flow. A higher complexity means more paths to test. Limit V(G) to < 10.

      Roman Urdu Explanation: Yeh code ki complexity ko measure karta hai. Jitne zyada loops ya conditions honge, complexity utni hi barhegi. Isko 10 se kam rakhna chahiye.

      3. Rework Ratio

      $$\text{Rework Ratio} = \frac{\text{Rework Hours}}{\text{Total Effort Hours}} \times 100$$

      What Symbols Mean: Rework Hours = Hours spent fixing bugs, refactoring, or re-doing bad tasks.

      Where to Use & Explanation: Analyzes process waste. If rework ratio is > 15%, the development process suffers from poor requirements or coding quality.

      Roman Urdu Explanation: Yeh batata hai ke total time me se kitna time ghaltiyan theek karne me zaya hua. Isko 10-15% se kam hona chahiye.

      4. Mean Time Between Failures (MTBF)

      $$MTBF = MTTF + MTTR$$

      What Symbols Mean: MTTF = Mean Time to Failure (uptime), MTTR = Mean Time to Repair (downtime/fixing duration).

      Where to Use & Explanation: Measures software reliability. Shows the average time between two consecutive failures.

      Roman Urdu Explanation: Do failures ke darmiyan average time kitna tha. Yeh jitna zyada hoga, system utna hi reliable hoga.

      5. System Availability

      $$\text{Availability} = \frac{MTBF - MTTR}{MTBF} \times 100$$

      What Symbols Mean: MTBF = Mean Time Between Failures, MTTR = Mean Time to Repair, MTTF = Mean Time to Failure.

      Where to Use & Explanation: Used to assess service availability percentage. Goal is three-nines (>99.9%) or four-nines uptime.

      Roman Urdu Explanation: System kitne percent chal raha tha aur useable tha. 99.9% availability industry standard hai.

      6. Basic COCOMO Estimation

      $$\text{Effort} = a \times (\text{KLOC})^b \quad \text{and} \quad T_{dev} = c \times (\text{Effort})^d$$

      What Symbols Mean: KLOC = Thousands of lines of code, a, b, c, d = Complexity constants (Organic, Semi-detached, Embedded).

      Where to Use & Explanation: Used in early project planning to estimate the total person-months and calendar time required for software development.

      Roman Urdu Explanation: KLOC ki madad se project me lagne wali effort (person-months) aur time (months) estimate kiya jata hai.

      7. Programmer Productivity (PUM)

      $$\text{PUM} = \frac{\text{LOC}}{\text{Person-Month}}$$

      What Symbols Mean: LOC = Lines of Code produced, Person-Month = Developer hours unit.

      Where to Use & Explanation: Measures productivity volume, but should not be used as a sole measure of code quality.

      Roman Urdu Explanation: Ek developer ek mahine me kitni lines of code likhta hai. Yeh quality nahi balkay sirf volume batata hai.

      8. Defect Density

      $$\text{Defect Density} = \frac{\text{Defects}}{\text{KLOC}}$$

      What Symbols Mean: Defects = Number of bugs discovered, KLOC = Thousands of lines of code.

      Where to Use & Explanation: Evaluates overall product quality. Higher density points to potential design and coding issues.

      Roman Urdu Explanation: Code ke size ke hisab se kitne defects nikle. Yeh density jitni kam hogi, code utna hi behtar aur reliable hoga.

      9. Test Case Effectiveness (TCE)

      $$TCE = \frac{\text{Defects found by tests}}{\text{Total defects found}} \times 100$$

      What Symbols Mean: Defects found by tests = Bugs caught during test executions, Total defects found = Total bugs (tests + customer bugs).

      Where to Use & Explanation: Assess whether your test suite is designed well enough to capture potential bugs before release.

      Roman Urdu Explanation: Humari test cases ne kitne percent defects dhonde. Yeh value jitni high hogi, testing quality utni achi hai.
